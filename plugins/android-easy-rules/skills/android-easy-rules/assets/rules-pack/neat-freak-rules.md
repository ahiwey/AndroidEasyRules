# Neat Freak Rules

本规则融合自 `KKKKhazix/khazix-skills/neat-freak`，用于知识治理和项目收尾。来源采用 MIT License；本规则只提炼可复用工作流，不引入外部脚本、evals 或自动清理动作。

## 触发条件

- 用户点名 `neat-freak`、`洁癖`、`/neat`。
- 用户要求收尾、同步文档、同步规则、整理记忆、检查规则是否过期、清点 workspace 残留。
- 开发完成后需要给下一次会话、同事或新 agent 一个当前可信入口。

不要因普通代码实现、纯重构、单纯整理 prose/JSON/changelog，或模糊一句“整理一下”就自动触发完整洁癖流程。

## 完成合同

收尾不是只看 `git status`，而是逐面说明事实状态。按任务范围选择以下事实面：

| 事实面 | 要回答的问题 | 常见证据 |
| --- | --- | --- |
| 代码 | 当前代码真正实现了什么 | 当前分支、配置、schema、测试、diff |
| 运行态 | 用户实际能访问到什么 | 构建产物、部署标记、页面/API、日志 |
| 文档 | README/docs 是否是现役答案 | README、架构、接入、运维、索引 |
| 规则 | Agent 规则是否同源、可执行、无死引用 | `AGENTS.md`、`CLAUDE.md`、模块规则、hooks |
| 记忆 | 长期记忆是否准确且允许修改 | 平台记忆入口、索引、生成来源 |
| 工作区 | 是否有未审计残留 | 临时计划、调试脚本、旧副本、worktree |

每个适用事实面标记为 `verified-current`、`changed-and-verified`、`pending`、`out-of-scope` 或 `not-applicable`。无法验证就标 `pending`，不要把猜测写进权威文档。

## 路径选择

### 轻量路径

适用于单人项目、规则少、无发布/远端协作、无多项目联动的普通收尾。

1. 盘点：列出根目录、Markdown、规则文件、入口配置和当前 git 状态，跳过依赖与构建目录。
2. 对齐事实：以当前代码、配置和可运行证据为准修正文档或规则中的过期说法。
3. 补规则：有可运行代码但没有项目规则时，创建最小 `AGENTS.md`；已有规则只修矛盾和死引用。
4. 列残留：一次性计划、调试脚本、旧副本和替代文件只列删除候选；未获确认前不删除。
5. 汇报：说明影响、改动、验证、待确认清理项和仍为 `pending` 的事实面。

### 完整路径

任一命中时走完整路径：有部署/发布面、多项目联动、远端协作、workspace 审计、项目规则要求发布收尾，或用户明确要求“完整洁癖”。

- 建立事实矩阵：记录权威来源、过期表面、拟采取动作和验证方式。
- 审计规则链：检查上下级 `AGENTS.md`、`CLAUDE.md`、override、导入、软链、hooks 是否冲突或死引用。
- 路由知识面：代码/API/schema/配置/用户流程变化，要同步到现有 README、docs、规则或 MEMORY，不强造固定文件。
- 先减后加：删除或改写过期现役说法，历史过程归 git/changelog/事故文档；规则层只留未来还会犯错的约束。
- 谨慎处理记忆：没有用户或平台明确授权时，记忆只读；机器生成记忆不手改。

## 清场边界

- 删除分支、worktree、临时库、中间产物、旧副本或调试脚本属于清场，不在最终汇报前自动执行。
- 默认先给出待删除候选、理由和风险，保留复核现场；用户看完汇报后明确确认，才能删除。
- 用户一开始说“做完清理”不等于看完结果后的二次确认。
- 清场后必须重新检查工作区，并补充实际删除项、残留 warning 和仍未关闭的 `pending`。

## 报告格式

洁癖收尾结果优先按以下结构汇报：

```text
## 洁癖收尾完成

影响：<消除了哪些误导、风险或交接成本>

改动 / 新建
- <文件> — <改了什么，为什么>

验证
- <命令或检查>：<结果>

待你确认
- 删除候选：<文件 + 理由>；未确认前一个都不删
- 无法裁决：<矛盾 + 两边证据>

遗留
- <pending / out-of-scope / warning>
```

没有部署、记忆或发布面时，明确标为 `not-applicable`，不要硬凑证据。
