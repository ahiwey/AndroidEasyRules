# Screenshot Driven UI Rules

## 硬规则

- 凡是截图、效果图或 UI 设计图驱动的任务，必须先视觉解析，再实现，最后说明截图验证或编译验证结果。
- 没有 Figma 时，把截图/效果图作为主要设计依据，但不要凭文字描述猜视觉细节。
- 新页面、整体重设计、跨屏交互或体验评审优先使用 `ui-ux-pro-max`；明确控件的间距、颜色、尺寸、裁切和 Canvas 参数小改直接走 Quick，避免为局部问题加载通用设计流程。

## 视觉解析清单

- 页面结构和信息层级
- 关键间距和对齐方式
- 主色、背景色、状态色
- 字号层级、字重、行高
- 圆角、阴影、边框、分割线
- 图标尺寸、透明边距和状态差异
- 可点击区域和触控尺寸
- 空数据、加载态、错误态
- 顶部状态栏、底部导航、安全区域
- 小屏和长文本风险

## 实现流程

1. 查看用户提供的截图/图标/图片。
2. 新页面、整体重设计或实现方式不明确时查找相似页面、资源和自定义 View；明确控件的局部问题只读取目标实现和实际使用位置。
3. 确定使用 XML layout、drawable、bitmap、custom View 或 adapter 的实现方式。
4. 实现时优先复用项目已有 style、colors、dimens、drawable 和资源命名。
5. 只有新增/重命名资源或引用、且静态检查无法覆盖时，才运行一次最小资源链接任务；现有属性值调整不默认运行 Gradle。
6. 能跑模拟器/真机时，截图核对目标图。
7. 最终回复说明验证方式和未验证项。

## 禁止项

- 不要把整张截图或未裁切设计稿直接放进 `res`。
- 不要为了像截图而硬编码文案、尺寸或颜色到 Kotlin/Java。
- 不要让文字、图标或按钮在小屏上重叠。
- 不要用拉伸 bitmap 解决布局问题。
- 不要忽略空数据、加载态和长文本。
